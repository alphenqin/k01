# K01 自动研判与数据分析需求文档（草案）

## 1. 背景与目标

基于输入文件 `/mnt/d/projects/k01/input/input.xlsx` 中的外联 IOC 数据，结合 xmon、文件情报、wfy、sc、wd 等接口结果及一系列判断规则，自动生成两个 Excel：

- `/mnt/d/projects/k01/output/result.xlsx`：K01 研判结果导入表，面向最终导入或交付。
- `/mnt/d/projects/k01/output/数据分析.xlsx`：覆盖度与可解决性分析表，面向规则核对、人工复盘和后续优化。

当前需求重点不是一次性把规则完全定死，而是先形成一版可执行、可核对、可逐步补充的需求基线。

## 3. 外部接口与查询数据

### 3.1 xmon IOC 查询

用途：

- 查询 IOC 在 xmon 中的主线索、子线索、禁用状态、存活状态、关联样本、关联报告等信息。

第一道接口：主线索查询

- URL：`http://xmon.netlab.qihoo.net/api/iocmon-search/ioc/{ioc_list}/?ui_simple=true&pretick=false&model=false&trace=true&keep_no=false&inspect=true&other_source=false`
- 方法：`GET`
- 请求头：
  - `X-AuthToken: 6189ff22-21c8-49ab-8519-7ef85f396954`
  - `Referer: http://xmon.netlab.qihoo.net/ui/iocmon/`
  - `Cookie: XMON-User=...`，如当前环境要求登录态则需要携带；脚本中通过 `K01_XMON_COOKIE` 环境变量配置。

返回结构：

- 外层 `code = 0` 且 `msg = success` 表示请求成功。
- IOC 结果在 `data` 数组中，一条 IOC 通常对应一个对象。
- 支持批量查询，多个 IOC 在同一个路径段中用英文逗号 `,` 分隔。例如：
  - 单个：`/api/iocmon-search/ioc/96.92.242.30:80/?...`
  - 批量：`/api/iocmon-search/ioc/96.92.242.30:80,8.8.8.8:53/?...`
- 批量结果只返回命中的 IOC；未返回的 IOC 按未查到处理。

第二道接口：子线索查询

- 入参来自第一道接口返回的 `data[].ioc`，不是原始查询 IOC。
- URL：`http://xmon.netlab.qihoo.net/api/tagmon-search/ioc/{ioc}/limit/1000/?data_only=true`
- 方法：`GET`
- 请求头与第一道接口一致。
- 返回结构同样为外层 `code = 0`、`msg = success`、`data` 数组。
- 子线索筛选规则：
  - 子线索 `data[].ioc` 必须与第一道主线索 `data[].ioc` 完全一致，才认为两者有主子线索关系。
  - 子线索 `ioc` 与主线索 `ioc` 不一致时忽略。
  - 主线索只看 `ioc_disabled` 字段；该字段非空时排除。
  - 子线索只看 `exts.disabled` 字段；该字段非空时排除。
  - 主线索和子线索在生产方归属判断前统一合并成一个线索列表，不再区分主子关系。
  - 合并后的线索列表中，忽略 `severity <= 30` 的线索。
  - 合并后的线索列表中，主线索忽略 `ioc_disabled` 非空的线索；子线索忽略 `exts.disabled` 非空的线索。

返回示例关键片段：

```json
{
  "code": 0,
  "msg": "success",
  "data": [
    {
      "ioc": "96.92.242.30:80",
      "ioc_search": "96.92.242.30:80",
      "ioc_type": "ip:port",
      "ioc_disabled": [],
      "status": [
        "ACTIVE"
      ],
      "src": "apt.siyubo",
      "ref_hash_list": [],
      "ref_sample": [],
      "report_links": [],
      "severity": 70,
      "confidence": 90,
      "tag_main": {
        "family": "evilopenclawskills",
        "severity": 70,
        "label_zh": [
          "后门软件"
        ]
      },
      "tags_info": [
        {
          "src": "apt.siyubo",
          "family": "EvilOpenClawSkills",
          "disabled": ""
        }
      ],
      "ti_info": {
        "familys": [
          "EvilOpenClawSkills"
        ],
        "malicious_names": [
          "后门软件/Malware.Backdoor"
        ]
      }
    }
  ]
}
```

关键字段：

| 字段 | 用途 |
| --- | --- |
| `ioc` / `ioc_search` / `uid` | 匹配查询 IOC |
| `ioc_disabled` | 主线索 disabled 判断字段；空数组或空字符串表示未 disabled |
| `exts.disabled` | 子线索 disabled 判断字段；空字符串表示未 disabled |
| `status` | 存活状态，实际可能是数组，例如 `["ACTIVE"]` |
| `ref_sample` | 主线索关联样本 hash 来源；该字段为列表，取第一个对象的 `md5` 字段 |
| `exts._raw.md5` | 子线索关联样本 hash 来源 |
| `report_links` / `Raw.report_links` | 提取关联报告链接，可能是数组 |
| `src` | 主线索生产方归属字段，例如 `apt.siyubo` |
| `src` | 主线索和子线索生产方归属字段，例如 `apt.siyubo` |
| `tag_main` / `tags` / `tags_info` / `ti_info` | 辅助识别家族、恶意类型、生产方归属 |
| `severity` / `confidence` | 风险和置信度，可作为后续分析扩展字段 |

生产方归属使用的 xmon 数据口径：

- 先取第一道接口的主线索。
- 再用主线索 `ioc` 查询第二道接口的子线索。
- 将主线索和符合关系的子线索合并成一个大列表。
- 过滤掉 `severity <= 30` 和 disabled 非空的线索。
- 在过滤后的列表中，atateam / siyubo / netlab 只使用指定字段判断：主线索和子线索都看顶层 `src`。
- 主线索 `src` 或子线索 `src` 包含 `btmon` 时，将 `netlab` 加入生产方归属候选。
- 主线索 `src` 或子线索 `src` 为 URL 时，将 `netlab` 加入生产方归属候选。

存活状态映射：

| xmon status | K01 存活状态 |
| --- | --- |
| `ACTIVE` | 存活 |
| `UNKNOWN` | 失活 |
| `SINKHOLE` | 被安全机构接管 |

### 3.2 文件情报接口

用途：

- 对 xmon 关联样本 hash 调用文件情报接口。
- 判断样本是否黑。
- 补充文件大小、文件类型、影响操作系统、创建时间、恶意家族等字段。

关键判断：

| 字段 | 规则 |
| --- | --- |
| `risk in {critical, high, medium, low}` | 认为关联样本为黑样本，可作为有效证据 |

| `risk = unkown` / `unknown` / 无结果 | 未知，不命中黑样本规则 |

### 3.3 wfy reputation 接口

用途：

- 判断 IOC 是否被 wfy 识别为黑。
- 作为生产方归属深度判断的重要前置条件。
- 在 `result.xlsx` 的补充信息规则中，wfy 白名单优先；wfy black 后再按黑样本、关联报告、wd 快照、siyubo 证据链、智能体证据链处理；wfy 无结果 / unknown 作为兜底，补充信息填 `wfy接口查询显示未知`。

接口：

- URL：`http://api.netlab.360.com/wfy/reputation/v1`
- 方法：`POST`
- 请求头：
  - `X-AuthToken: 6189ff22-21c8-49ab-8519-7ef85f396954`
  - `Content-Type: application/json`
- 请求体：IOC 字符串数组，例如：

```json
[
  "96.92.242.30:80"
]
```

执行策略：

- 脚本不再使用 100 条批量查询。
- 为避免批量返回漏 IOC，wfy 按单 IOC 查询，每次请求体只放 1 个 IOC，并使用线程池并发执行。

关键返回字段：

| 字段 | 说明 |
| --- | --- |
| `data.{ioc}.judge` | 黑白判定，`black` 表示 wfy 判黑 |
| `data.{ioc}.status` | wfy 侧存活状态，例如 `ACTIVE` |
| `data.{ioc}.risk_level` | 风险等级 |
| `data.{ioc}.malicious_family` | 恶意家族 |

| wfy 无结果 / `data` 无结果 | 生产方归属直接填 `unknown`；若黑样本、关联报告、wd 快照均未命中，则补充信息填 `wfy接口查询显示未知`，`相关解决方案` 填 `wfy查不到` |
| `judge = unknown` 或其他非 black | 生产方归属不强制 `unknown`，继续按 xmon / wd / sc 候选判断 |

补充信息模板：

`wfy接口查询显示未知`

`wfy未报告恶意`

### 3.4 sc 接口

用途：

- 辅助生产方归属判断。
- 若 sc 查询结果为恶意，且未命中 atateam、siyubo、wd，则默认归属 netlab。

接口：

- URL：`https://api.ti.360.net/custom/intelligence/v1/tags`
- 方法：`POST`
- 请求头：
  - `Content-Type: application/json`
  - `X-Api-Key: api_key`
  - `sign: sign`
  - `timestamp: timestamp`
- 签名方式：与文件情报接口一致，`sign = md5(str(timestamp) + SALT)`。

请求体：

```json
{
  "query": {
    "keywords": [
      {
        "field": "category",
        "value": "domain"
      },
      {
        "field": "query",
        "value": "223.200.120.73:443"
      },
      {
        "field": "flag",
        "value": "2"
      }
    ]
  }
}
```

说明：

- 脚本当前输入侧只会生成域名或 `IP:端口` 形式的 IOC，这两类查询 SC 时 `category` 均传 `domain`。
- 如果后续输入为完整 `http://` / `https://` URL，才传 `category=url`。

关键返回字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `data.level` | int | 危害等级，0-100 的整十数分级；30 为未知，30 以下为白，30 以上为黑，30-50 为低危，50-60 为中危，60-70 为高危，70 以上为关键 |

恶意判定：

| 条件 | 处理 |
| --- | --- |
| `data.level >= 50` | 认为 sc 报恶意；若未命中 atateam、siyubo、wd，则生产方归属为 `netlab` |
| `data.level < 50` 或无 `level` | 不作为恶意证据，按未查到处理 |

### 3.5 wd / 内容快照智能体

用途：

- 判断 IOC 是否存在恶意快照。
- 对 `wd` 生产方归属提供证据。

#### 3.5.1 wd 评分接口

来源脚本：`/mnt/d/projects/k01/wd/wd_v1_now.py`

接口：

- URL：`http://api.safe.qihoo.net/urls/std/v1/cloud_safe_info`
- 方法：`POST`
- 请求头：
  - `X-360-Key`
  - `X-360-Nonce`
  - `X-360-Timestamp`
  - `X-360-Signature`
  - `Content-Type: application/json`
- 签名方式：
  - `body_md5 = md5(body)`
  - `signature_source = body_md5 + appid + nonce + timestamp + secret`
  - `X-360-Signature = md5(signature_source)[16:]`
- 请求体：

```json
{
  "data": [
    {
      "url": "example.com"
    }
  ]
}
```

关键返回字段：

| 字段 | 说明 |
| --- | --- |
| `data.results[0].info.level` | wd 评分 |
| `data.results[0].info.sub_level` | wd 子评分 |

恶意判定：

| 条件 | 处理 |
| --- | --- |
| `level >= 50` | wd 评分认为 IOC 恶意 |
| `level < 50` 或无结果 | 不作为 wd 恶意证据 |

#### 3.5.2 wd 历史 HTML 快照接口

来源脚本：`/mnt/d/projects/k01/wd/input2html_history.py`

接口：

- URL：`http://api.netlab.qihoo.net/urldb/v0/detail/`
- 方法：`GET`
- 请求头：
  - `X-Authtoken`
- 查询参数：
  - `query={ioc}`
  - `time_start=2020-05-29 00:00:00`

有效快照判定：

| 条件 | 处理 |
| --- | --- |
| 返回 `data` 为列表，且存在至少一条记录包含非空 `html` 或非空 `snapshot` 字段 | 认为存在有效快照 |
| 无 `data`、`data` 为空、记录无 `html` 且无 `snapshot` | 认为没有有效快照 |

主题内容提取：

- 优先取快照记录的 `title`。
- `title` 为空时，回退取 `url` / `domain` / IOC。

#### 3.5.3 wd 归属与可解决规则

- 只有生产方归属优先级判断到 `wd` 时，才使用 wd 结果。
- 同时满足以下条件，生产方归属填 `wd`：
  - wfy 判黑；
  - 未命中更高优先级的 `atateam`、`siyubo`；
  - wd 评分 `level >= 50`；
  - wd 历史 HTML 快照有效。
- 命中 wd 规则时，补充信息模板为：`内容类，存在恶意快照的ioc({主题内容})`。
- 命中 wd 规则时，`能否解决` 填 `能`。

### 3.6 siyubo 证据链与智能体证据链

用途：

- 当未命中黑样本、关联报告、wd 快照，且生产方归属为 `siyubo` 时，补充情报研判依据。

siyubo 子线索证据链：

- 来源：xmon 子线索字段 `exts._raw.ext.evidence_chain`。
- 仅在生产方归属为 `siyubo` 时使用。
- 若 `evidence_chain` 为非空列表，提取其中每个对象的 `detail` 字段。
- 将 `detail` 汇总为 50 字以内的情报研判依据，填入 `补充信息`。
- 对应 `命中规则`：`siyubo证据链`。

智能体证据链接口：

- 触发条件：生产方归属为 `siyubo`，且未命中黑样本、关联报告、wd 快照、siyubo 子线索证据链。
- URL：`http://api.netlab.qihoo.net/ai/quick-analysis`
- 方法：`POST`
- 请求头：
  - `Content-Type: application/json`
  - `X-AuthToken: 6189ff22-21c8-49ab-8519-7ef85f396954`
- 请求体：

```json
{
  "ioc": "ttcxdljiue.cn",
  "ioc_type": "domain"
}
```

关键返回字段：

| 字段 | 说明 |
| --- | --- |
| `data.key_evidence` | 智能体输出的关键证据列表 |

处理规则：

- 若 `key_evidence` 非空，过滤掉包含 `外部威胁情报显示` 的证据行。
- 将剩余证据汇总为 50 字以内话术，填入 `补充信息`。
- 对应 `命中规则`：`智能体证据链`。
- 若 `key_evidence` 为空或不存在，则继续走后续 wfy unknown / 无更多证据规则。
- 智能体接口失败时进行有限重试：最多重试 3 次，按 2、4、6 秒递增等待；最终失败只进入控制台失败摘要，不中断整体脚本。
- 为减少 500 / 502 / 超时，智能体接口默认并发数控制为 8。

## 4. 输出一：result.xlsx

| 误报原因 | 无效、白、误报时填写原因；当前规则待确认 |
| 命中规则 | 与补充信息对应的固定规则名称 |

### 4.3 研判结果规则

按优先级从上到下判断，命中后可直接产出结果：

| 优先级 | 规则 | 研判结果 | 说明 |

| 4 | 生产方归属为 wd，且 wd 评分恶意、存在有效快照 | 有效 | 输出内容快照证据；只有生产方归属为 wd 时才使用 wd 快照模板 |
| 5 | 生产方归属为 siyubo，且 xmon 子线索存在 evidence_chain | 有效 | 汇总 evidence_chain.detail 为 50 字以内依据 |

| 9 | 文件情报为白或未命中更多证据 | 空 | 不生成补充信息 |

### 4.4 补充信息生成规则

补充信息只能使用固定模板，且每行最多只填一种。按以下优先级命中后立即停止：

| 命中规则 | 补充信息模板 |
| --- | --- |
| wfy 白 | `wfy未报告恶意` |
| 关联黑样本 | `{ioc}，依据ioc({ioc}),关联样本（{hash}）` |
| 关联报告 | `{ioc}，依据ioc({ioc}),关联报告（{report_url}）` |
| wd 恶意快照 | `内容类，存在恶意快照的ioc({主题内容})` |
| siyubo 证据链 | evidence_chain.detail 汇总后的 50 字以内话术 |
| 智能体证据链 | key_evidence 汇总后的 50 字以内话术 |
| 生产方统计话术 | `{生产方归属}量有{总量}，解决了{能解决量}，{不能解决量}没解决` |
| wfy 无结果 / unknown | `wfy接口查询显示未知` |

命中规则列固定取值：

- `wfy未报告恶意`
- `存在黑样本关联`
- `存在关联报告关联`
- `src是wd且有快照`
- `siyubo证据链`
- `智能体证据链`
- 空：生产方统计话术、wfy unknown / 无结果、无更多证据等场景

生产方统计话术：

- 触发位置：紧接在智能体证据链之后。
- 触发条件：wfy `judge = black`，且黑样本、关联报告、wd 快照、apt.siyubo 证据链、智能体证据链均未命中。
- 统计范围：本次已经生成的全部 `数据分析.xlsx` 数据。
- 统计字段：
  - `生产方归属`：按 `atateam`、`siyubo`、`wd`、`netlab`、`unknown` 统计。
  - `能否解决`：值为 `能` 计入解决量，其他值计入未解决量。
- 填补顺序按生产方归属优先级：`atateam`、`siyubo`、`wd`、`netlab`、`unknown`。
- 话术模板：`{生产方归属}量有{总量}，解决了{能解决量}，{不能解决量}没解决`。
- 生产方统计话术只填 `补充信息` 和 `相关解决方案`，`result.xlsx` 的 `命中规则` 保持为空。

多条证据处理：

- 关联样本 hash 提取规则：收集所有有效主线索和子线索 hash；主线索取 `ref_sample` 列表第一个对象的 `md5` 字段，子线索取 `exts._raw.md5`。所有 hash 先去重，再查询文件情报，然后从 `risk in {critical, high, medium, low}` 的结果中选择一个填入表。
- 文件情报返回结果解析时，优先按请求 hash 匹配 `result` 的 key；同时兼容 key 大小写差异，并可用返回体内的 `md5` / `sha1` / `sha256` 字段反向匹配，避免接口返回 key 形式变化导致 `risk` 被误判为空。
- 文件情报不再使用批量 hash 查询；所有 hash 去重后按单 hash 并发查询，避免批量接口漏返回导致黑样本证据丢失。
- 关联报告可能有多个，当前暂取第一个。
- 不拼接多条补充信息。

### 4.5 关联报告链接规则

- 关联报告链接来自 xmon 的 `report_links` 字段。
- 将字段按列表、JSON、逗号、分号、空白等形式拆分。
- 当前规则不做域名白名单过滤，取拆分后的第一个非空链接。
- 如果链接中包含 `#` 片段，保留 `#` 前的主链接。

## 6. 生产方归属逻辑

### 6.1 总体流程

1. 查询 xmon 主线索和子线索，作为黑样本、关联报告和归属关键词来源。
2. 查询文件情报接口，判断 xmon 关联样本是否为黑样本。
3. 查询 wfy 接口。
4. 如果 wfy 接口没有查到有效数据，例如 `data` 没有结果，生产方归属直接填 `unknown`。
5. 如果 wfy 有有效返回，不管 `judge` 是 black、white、unknown 还是其他值，都将 xmon src、wd、sc 等来源识别到的生产方归属统一放入候选列表。
6. 最终生产方归属从候选列表中按 `atateam`、`siyubo`、`wd`、`netlab`、`unknown` 的顺序选择一个。
7. wd 只有在评分恶意且存在有效快照时，才加入候选列表。
8. 基于 `result.xlsx` 的补充信息模板生成 `能否解决` 与 `相关解决方案`。

### 6.2 归属优先级

建议按以下顺序判定：

| 优先级 | 归属 | 判定规则 |
| --- | --- | --- |
| 1 | atateam | xmon 主线索 `src` 包含 `atateam`，或子线索 `src` 包含 `atateam` |
| 2 | siyubo | xmon 主线索 `src` 包含 `siyubo`，或子线索 `src` 包含 `siyubo` |
| 3 | wd | wd 评分 `level >= 50` 且历史 HTML 快照有效 |
| 4 | netlab | sc 报恶意；或 xmon 主线索 `src` / 子线索 `src` 包含 `btmon`；或该 src 字段为 URL |
| 5 | unknown | wfy 不报黑、无结果，或当前来源无法识别 |

### 6.3 xmon 线索有效性

当前判断口径：

- 主线索：`ioc_disabled` 为空，且 `severity > 30`，认为可用；`ioc_disabled` 非空或 `severity <= 30` 时排除。
- 子线索：子线索 `ioc` 必须等于主线索 `ioc`；`exts.disabled` 为空，且 `severity > 30`，认为可用；`exts.disabled` 非空或 `severity <= 30` 时排除。
- atateam / siyubo / netlab 归属只看可用线索中的主线索 `src` 和子线索 `src`；其中 `btmon` 或 URL 型 src 映射为 `netlab`。
- 如果点数数量在 `[0,1]`，可能是内部情报。
- 其他情况可能是开源情报。

待确认：

- “点数数量”的字段名。
- 主线索、子线索的接口字段名。
- “内部情报”和“开源情报”是否影响生产方归属，还是只影响可解决性。
